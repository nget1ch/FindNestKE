📱

M-Pesa STK Push

Safaricom sends PIN prompt to phone. On success, callback hits /api/payments/mpesa/callback with receipt number.

💳

Stripe Card Payment

Frontend confirms via Stripe.js, then calls /api/payments/card/confirm to verify intent status server-side.

🔒

Transactional Outbox

Booking update, payment record, and compliance job are all committed in a single atomic DB transaction. If any fails, none persist.

🏛️

GavaConnect / KRA eTIMS

Tax calculations (MRI 7.5%, VAT 16%, Tourism Levy 2%) are sent to the KRA Sandbox. Results logged to compliance_logs table.

Diagram 06

Background Job Worker Lifecycle

Persistent job queue with exponential backoff retry logic. Ensures every payment eventually generates a KRA compliance record, even during API outages.

flowchart TD
    START["🚀 Server Start\nrunWorker()"]
    POLL["⏲️ Poll every 30 seconds\nSELECT FROM jobs\nWHERE status = pending\nAND nextRunAt < NOW\nLIMIT 5"]
    LOCK["🔒 Set status → processing"]
    PROCESS{"🔀 Job Type?"}
    
    ETIMS["🏛️ kra_etims_sync\n──────────\nsendRevenueToGava(payload)\nCalculate MRI + VAT\nPOST to KRA Sandbox"]
    
    SUCCESS["✅ Job Completed\nstatus → completed"]
    
    FAIL["❌ Exception Caught"]
    INCREMENT["📈 attempts += 1"]
    
    CHECK{"attempts >=\nmaxAttempts (5)?"}
    
    BACKOFF["🔄 Exponential Backoff\nnextRunAt = now + 2^n × 60s\nstatus → pending"]
    
    DEAD["💀 Permanently Failed\nstatus → failed\nAdmin notification triggered"]
    
    RETURN["↩️ Return to poll loop"]

    START --> POLL
    POLL --> LOCK
    LOCK --> PROCESS
    PROCESS -->|"kra_etims_sync"| ETIMS
    ETIMS -->|"Success"| SUCCESS --> RETURN
    ETIMS -->|"Error"| FAIL
    FAIL --> INCREMENT --> CHECK
    CHECK -->|"No"| BACKOFF --> RETURN
    CHECK -->|"Yes"| DEAD
    RETURN --> POLL

⏱️

Retry Schedule

Attempt 1: 2 min wait · Attempt 2: 4 min · Attempt 3: 8 min · Attempt 4: 16 min · Attempt 5: permanent failure.

💀

Dead Letter

After 5 failed attempts, the job is marked permanently failed. Admin is notified via the dashboard to manually investigate.

🔒

Idempotency

generateETIMSReceipt checks for existing non-rejected compliance logs before creating duplicates, preventing double receipts.
